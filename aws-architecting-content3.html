Amazon EC2 Auto Scaling

Auto Scaling Purpose

  • Dynamic Capacity: Correct number of instances for load
  • Cost Optimization: Scale out when needed, scale in when not
  • High Availability: Replace unhealthy instances automatically

Auto Scaling Components

Three Key Components

Launch Template â†’ Auto Scaling Group â†’ Scaling Policies

Component Functions

  • Launch Template: Instance configuration blueprint
  • Auto Scaling Group: Capacity management and placement

Auto Scaling Groups

ASG Configuration

  • Desired Capacity: Target number of instances
  • Minimum Capacity: Lower limit for scaling
  • Maximum Capacity: Upper limit for scaling
  • Availability Zones: Instance distribution across AZs

Launch Templates vs Launch Configurations

Feature Launch Template Launch Configuration
Versioning Supported Not supported
Mixed Instance Types Yes No
Spot Instances Advanced support Basic support

Launch Template Advantages

Modern Features

  • Versioning: Track and manage template changes
  • Parameter Inheritance: Override specific parameters
  • T2/T3 Unlimited: Burst performance configuration
Best Practice: Always use Launch Templates for new Auto Scaling implementations

Auto Scaling Policies

Policy Types

  • Target Tracking: Maintain specific metric value
  • Step Scaling: Scale in steps based on metric thresholds
  • Simple Scaling: Single scaling action per alarm
  • Scheduled Scaling: Predictable scaling patterns

Target Tracking Scaling

How Target Tracking Works

  • Set Target Value: e.g., 50% CPU utilization
  • Automatic Scaling: AWS manages scale-out and scale-in
  • CloudWatch Integration: Uses CloudWatch metrics

Target Tracking Metrics

Metric Use Case Target Value Example
CPU Utilization General workloads 50%
Request Count Per Target Web applications 1000 requests
Network In/Out Network-intensive apps 10 MB/s

Auto Scaling Best Practices

Configuration Guidelines

  • Health Checks: Use both EC2 and ELB health checks
  • Cooldown Periods: Prevent rapid scaling oscillations
  • Multiple AZs: Distribute instances for high availability

Cooldown and Warmup Periods

Timing Controls

Scale Event â†’ Cooldown â†’ Next Scale Event

Period Purposes

  • Cooldown: Wait time after scaling activity (300 seconds default)
  • Warmup: Time for instance to contribute to metrics

Amazon CloudWatch Overview

CloudWatch Services

  • Metrics: Performance data from AWS services
  • Logs: Centralized log management
  • Events: System events and automated responses
  • Alarms: Notifications and automatic actions

CloudWatch Metrics

Metric Characteristics

  • Variables to Monitor: CPU, NetworkIn, DiskReadOps
  • Namespaces: Logical grouping (AWS/EC2, AWS/RDS)
  • Dimensions: Metric attributes (InstanceId, DBName)
  • Timestamps: Data points with time stamps

CloudWatch Service Integration

Native Integration

CloudWatch integrates with 70+ AWS services including EC2, DynamoDB, S3, Lambda, API Gateway

Monitoring Types

  • Basic Monitoring: 5-minute intervals (free)
  • Detailed Monitoring: 1-minute intervals (additional cost)

Custom CloudWatch Metrics

Creating Custom Metrics

  • Applications: Send metrics using AWS SDK
  • Custom Namespaces: Create your own metric categories
  • High Resolution: Up to 1-second intervals

CloudWatch Dashboards

Dashboard Features

  • Global Visibility: Dashboards available across regions
  • Cross-Account: Include metrics from different accounts
  • Auto-Refresh: 10s, 1m, 2m, 5m, 15m intervals
  • Sharing: Public access without AWS credentials

CloudWatch Alarms

Alarm Architecture

Metric â†’ Threshold â†’ Alarm State â†’ Action

Alarm States

  • OK: Metric is within threshold
  • ALARM: Metric breached threshold

CloudWatch Alarm Actions

Available Actions

  • Auto Scaling: Add or remove EC2 instances
  • EC2 Actions: Stop, terminate, reboot instances
  • SNS Notifications: Send alerts via email/SMS
  • Systems Manager: Execute automation documents

CloudWatch Logs

Log Sources

  • Elastic Beanstalk: Application logs
  • ECS: Container logs
  • Lambda: Function execution logs
  • VPC Flow Logs: Network traffic logs

CloudWatch Logs Features

Log Management

  • Log Groups: Logical grouping of log streams
  • Log Streams: Sequence of log events
  • Retention Policies: Control log storage duration
  • Metric Filters: Extract metrics from log data

Serverless Computing with AWS Lambda

Serverless Benefits

  • No Server Management: AWS handles infrastructure
  • Automatic Scaling: Scales with demand automatically
  • Pay-per-Execution: Only pay for compute time used

AWS Lambda Fundamentals

Function Characteristics

  • Runtime Support: Python, Node.js, Java, .NET, Go, Ruby
  • Execution Time: Maximum 15 minutes per invocation
  • Memory: 128 MB to 10,240 MB
  • Concurrent Executions: Up to 1,000 by default

Lambda Event Sources

Category Services Trigger Type
Storage S3, DynamoDB Object/Item changes
API API Gateway, ALB HTTP requests
Streaming Kinesis, SQS Message/record processing

Infrastructure as Code (IaC)

IaC Principles

  • Declarative: Define desired end state, not steps
  • Version Control: Track infrastructure changes
  • Repeatable: Consistent deployments across environments

Infrastructure as Code Benefits

Key Advantages

  • Consistency: Eliminate configuration drift
  • Speed: Rapid environment provisioning
  • Documentation: Infrastructure as living documentation
  • Testing: Test infrastructure changes safely

Terraform vs CloudFormation

Feature Terraform CloudFormation
Cloud Support Multi-cloud AWS only
Language HCL (HashiCorp) JSON/YAML
State Management State file AWS managed

Terraform Fundamentals

Core Concepts

  • Providers: Plugin that interfaces with APIs (AWS, Azure)
  • Resources: Infrastructure components (EC2, S3)
  • Data Sources: Read existing infrastructure
  • Modules: Reusable configuration packages

Terraform Workflow

Standard Workflow

terraform init â†’ terraform plan â†’ terraform apply

Workflow Steps

  • init: Initialize working directory
  • plan: Preview changes

Essential Terraform Commands

Command Purpose Use Case
terraform init Initialize directory First run, new providers
terraform plan Preview changes Before applying changes
terraform apply Apply changes Create/update resources
terraform destroy Destroy resources Clean up environment

Terraform State Management

State File Purpose

  • Resource Tracking: Maps configuration to real resources
  • Performance: Caches resource attributes
  • Collaboration: Shared state for team environments

Terraform Best Practices

Configuration Guidelines

  • Remote State: Store state in S3 with locking
  • Modules: Use modules for reusable components
  • Variables: Parameterize configurations
  • Version Control: Track all configuration files

Terraform Variables and Outputs

Variables

  • Input Variables: Parameterize configuration
  • Local Values: Computed values within configuration
  • Environment Variables: Set via TF_VAR_name

Outputs

  • Return Values: Export important resource attributes
