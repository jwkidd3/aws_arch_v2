AWS Global Infrastructure

Regions, Availability Zones, and Edge Locations

AWS Regions Overview

  • Current Scale: 33+ regions globally with 105+ availability zones
  • Geographic Areas: Separate geographic areas around the world
  • Data Sovereignty: Data stays within selected region unless explicitly moved
  • Service Availability: Not all services available in all regions
  • Compliance: Choose regions based on regulatory requirements
  • Latency: Select regions closest to your users

Region Selection Criteria

Technical Factors:

  • Latency: Distance to end users
  • Service Availability: Required services available
  • Feature Availability: Latest features first in US regions
  • Capacity: Sufficient capacity for workloads

Business Factors:

  • Compliance: Data residency requirements
  • Pricing: Regional pricing variations
  • Disaster Recovery: Geographic separation
  • Local Presence: Business operations location

Availability Zones Deep Dive

  • Physical Separation: Each AZ consists of one or more data centers
  • Isolated Infrastructure: Separate power, cooling, and networking
  • Interconnection: High-speed, low-latency fiber links between AZs
  • Distance: AZs separated by meaningful distance (miles apart)
  • Naming: Logical names (us-east-1a) mapped randomly per account
  • Minimum: Every region has at least 3 AZs for redundancy
  • Best Practice: Always architect for multi-AZ deployment

AZ Design Patterns

High Availability:

  • Deploy identical resources in multiple AZs
  • Use load balancers to distribute traffic
  • Configure auto failover mechanisms
  • Monitor health across all AZs

Disaster Recovery:

  • Primary/secondary AZ configuration
  • Data replication between AZs
  • Automated backup and restore
  • Regular disaster recovery testing

Edge Locations & Local Zones

Edge Locations:

  • Global Network: 600+ edge locations worldwide
  • CloudFront CDN: Content caching closer to users
  • Route 53: DNS resolution at the edge
  • AWS Global Accelerator: Application performance
  • Lambda@Edge: Serverless compute at edge

Local Zones:

  • Ultra-Low Latency: Single-digit millisecond latency
  • Location: In or near major metropolitan areas
  • Use Cases: Real-time gaming, media streaming
  • Services: EC2, EBS, VPC, ELB available
  • Extension: Extends parent region capabilities

Wavelength Zones

  • 5G Edge Computing: AWS infrastructure at telecom 5G networks
  • Ultra-Low Latency: Single-digit millisecond latency to mobile devices
  • Mobile Applications: AR/VR, autonomous vehicles, IoT
  • Carrier Partnership: Embedded in telecom provider networks
  • Seamless Integration: Same AWS APIs and tools

AWS Global Infrastructure Benefits

  • Global Reach: Deploy applications worldwide with consistent experience
  • High Availability: Multiple AZs provide redundancy and fault tolerance
  • Scalability: Add capacity in multiple locations as needed
  • Performance: Reduce latency with edge locations and local zones
  • Security: Physical security and compliance in all locations
  • Cost Optimization: Choose optimal regions for price/performance

Global Infrastructure Planning

Multi-Region Strategy:

  • Primary and secondary regions
  • Cross-region data replication
  • Geographic load distribution
  • Compliance with local regulations

Network Considerations:

  • Inter-region bandwidth costs
  • Data transfer charges
  • Regional peering connections
  • Content delivery optimization

Regional Services vs Global Services

Regional Services:

  • EC2 instances and EBS volumes
  • VPC and subnets
  • RDS and database instances
  • Lambda functions
  • Auto Scaling groups

Global Services:

  • CloudFront (Edge locations)
  • Route 53 (DNS)
  • IAM (Identity management)
  • CloudFormation templates
  • AWS Organizations

Infrastructure Resilience Patterns

  • Single AZ: Development and testing environments only
  • Multi-AZ: Production workloads with automatic failover
  • Multi-Region: Global applications and disaster recovery
  • Active-Active: Load distributed across multiple regions
  • Active-Passive: Primary region with standby for failover
  • Pilot Light: Minimal standby infrastructure for cost optimization

AWS Well-Architected Framework

Five pillars for building secure, reliable, efficient systems

Operational Excellence

  • Automation: Automate operations processes and procedures
  • Monitoring: Monitor systems to understand operational health
  • Continuous Improvement: Learn from failures and successes
  • Documentation: Document procedures and runbooks
  • Small Changes: Make frequent, small, reversible changes

Security

  • Identity Management: Strong identity foundation and least privilege
  • Defense in Depth: Apply security at all layers
  • Data Protection: Encrypt data in transit and at rest
  • Incident Response: Prepare for security events
  • Automation: Automate security best practices

Reliability

  • Fault Tolerance: System continues operating despite failures
  • Recovery Planning: Test disaster recovery procedures
  • Change Management: Monitor and manage changes to systems
  • Capacity Planning: Scale horizontally for resilience
  • Automation: Automate recovery from failures

Performance Efficiency

  • Right Sizing: Use appropriate resource types and sizes
  • Monitoring: Monitor performance metrics continuously
  • Trade-offs: Make informed architectural trade-offs
  • Evolution: Evolve architecture as technology advances
  • Experimentation: Use data to drive architectural decisions

Cost Optimization

  • Right Sizing: Select most cost-effective resources
  • Elasticity: Scale resources up and down as needed
  • Purchasing Options: Use Reserved Instances and Spot Instances
  • Monitoring: Track and analyze spending patterns
  • Lifecycle Management: Manage resource lifecycle efficiently

Well-Architected Design Principles

  • Design for Failure: Assume everything will fail and plan accordingly
  • Implement Elasticity: Scale out, not up, for better resilience
  • Decouple Components: Remove dependencies between system components
  • Loose Coupling: Use queues, load balancers, and service discovery
  • Security in Depth: Apply security at multiple layers
  • Think Parallel: Design systems to work in parallel, not sequentially

Well-Architected Assessment Process

Review Questions:

  • Each pillar has specific questions
  • High-risk issues (HRIs) identified
  • Medium-risk issues (MRIs) noted
  • Improvement recommendations provided

Improvement Plan:

  • Prioritize high and medium risks
  • Create action items with owners
  • Set timelines for implementation
  • Regular review and updates

Well-Architected Tools & Resources

  • AWS Well-Architected Tool: Self-service assessment in AWS Console
  • Well-Architected Lenses: Specialized assessments for specific workloads
  • Partner Programs: AWS Well-Architected Partner reviews
  • Workshops: Hands-on architecture review sessions
  • Documentation: Whitepapers and best practices guides
  • Architecture Center: Reference architectures and solutions

Common Architecture Anti-Patterns

Avoid These:

  • Single points of failure
  • Tight coupling between services
  • Lack of automation
  • Poor monitoring and logging

Better Approaches:

  • Redundancy and failover
  • Microservices architecture
  • Infrastructure as Code
  • Comprehensive observability

Cloud Native Architecture

Modern application design principles and patterns

What is Cloud Native?

  • Applications built specifically for cloud environments
  • Leverage cloud platform capabilities from day one
  • Consider hosting, services, and technologies holistically
  • Platform-aware design rather than infrastructure-agnostic
Traditional: Build application first, then figure out hosting
Cloud Native: Design with cloud services in mind

Cloud Native Benefits

  • Developer Productivity: Focus on business logic, not infrastructure
  • Business Agility: Rapid deployment, faster time-to-market
  • Scalability: Automatic resource provisioning based on demand
  • High Availability: Built-in redundancy and automatic failover
  • Cost Optimization: Pay-as-you-go model eliminates waste

Microservices Architecture

Decomposing applications into independently deployable services

Microservices Overview

  • Definition: Architectural style decomposing applications into small, autonomous services
  • Independence: Each service can be developed, deployed, and scaled independently
  • Business Focus: Services organized around business capabilities
  • Decentralized: Governance and data management distributed

Microservices Benefits

Technical:

  • Technology diversity
  • Independent scaling
  • Fault isolation
  • Easier testing

Organizational:

  • Team autonomy
  • Faster development
  • Parallel deployment
  • Clear ownership

Serverless Computing

Event-driven, scalable applications without server management

What is Serverless?

  • No Server Management: AWS manages the infrastructure
  • Event-Driven: Code runs in response to events
  • Automatic Scaling: Scales from zero to thousands of requests
  • Pay Per Use: Only pay for actual execution time
  • Stateless: Each execution is independent

AWS Serverless Services

  • AWS Lambda: Serverless compute for event-driven applications
  • API Gateway: Managed API service with built-in monitoring
  • DynamoDB: Serverless NoSQL database
  • S3: Serverless object storage with event triggers
  • EventBridge: Serverless event bus for application integration

Serverless Benefits & Use Cases

Benefits:

  • No infrastructure management
  • Automatic scaling
  • Cost effective for variable workloads
  • Built-in high availability

Use Cases:

  • API backends
  • Data processing
  • Real-time file processing
  • IoT backends

Amazon EC2

Elastic Compute Cloud - Virtual servers in the cloud

EC2 Instance Types

General Purpose:

  • T4g, T3, T2: Burstable
  • M6i, M5: Balanced
  • A1: ARM-based

Compute Optimized:

  • C6i, C5: High performance
  • C5n: Enhanced networking

Memory Optimized:

  • R6i, R5: High memory
  • X1e, X1: High memory

EC2 Pricing Models

On-Demand:

  • Pay by hour/second
  • No upfront costs
  • No commitments
  • Highest cost per hour

Reserved:

  • Up to 75% savings
  • 1-3 year commitment
  • Capacity guarantee
  • Steady workloads

EC2 Pricing Models (Continued)

Spot:

  • Up to 90% savings
  • 2-minute interruption notice
  • Fault-tolerant apps
  • Flexible timing

Dedicated:

  • Physical server isolation
  • Compliance requirements
  • Licensing constraints
  • Highest cost

Amazon Machine Images (AMI)

  • Template: Contains OS, applications, and configurations
  • AWS Marketplace: Pre-configured AMIs from vendors
  • Community AMIs: Shared by AWS community
  • Custom AMIs: Create your own standardized images
  • Regional: AMIs are region-specific resources

Elastic Block Store (EBS)

  • Persistent Storage: Block-level storage for EC2 instances
  • High Availability: Automatically replicated within AZ
  • Scalable: Modify volume size and type without downtime
  • Backup: Point-in-time snapshots stored in S3
  • Encryption: Encrypt data at rest and in transit

EBS Volume Types

SSD-based:

  • gp3/gp2: General Purpose
  • io2/io1: Provisioned IOPS
  • Better for random I/O
  • Can be boot volumes

HDD-based:

  • st1: Throughput Optimized
  • sc1: Cold HDD
  • Better for sequential I/O
  • Cannot be boot volumes

Identity and Access Management

Secure access control for AWS resources

IAM Core Components

  • Users: Individual people or services
  • Groups: Collections of users with common permissions
  • Roles: Temporary credentials for services or federated users
  • Policies: JSON documents defining permissions
  • Identity Providers: External authentication systems

IAM Best Practices

  • Least Privilege: Grant minimum permissions required
  • Root Account: Secure and avoid using for daily operations
  • Multi-Factor Authentication: Enable MFA for all users
  • Password Policy: Enforce strong password requirements
  • Regular Review: Audit permissions and remove unused access
  • Roles over Users: Use roles for applications and services

IAM Policy Structure

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}

Multi-Account Architecture

Strategic organization of AWS accounts for governance and security

Why Multi-Account Strategy?

  • Security Isolation: Strong boundary between environments and teams
  • Billing Separation: Clear cost allocation and chargeback models
  • Regulatory Compliance: Meet data isolation and governance requirements
  • Resource Limits: Avoid hitting service limits in single account
  • Blast Radius Reduction: Limit impact of security incidents or misconfigurations
  • Team Autonomy: Independent development and deployment cycles

AWS Organizations

Core Features:

  • Centralized account management
  • Consolidated billing
  • Service Control Policies (SCPs)
  • Organizational Units (OUs)
  • Account creation automation

Benefits:

  • Volume discounts across accounts
  • Centralized governance controls
  • Simplified compliance auditing
  • Consistent security policies
  • Resource sharing capabilities

Common Multi-Account Patterns

Environment-Based:

  • Development: Testing and experimentation
  • Staging: Pre-production validation
  • Production: Live customer workloads
  • Sandbox: Individual developer environments

Function-Based:

  • Security: Centralized logging and monitoring
  • Shared Services: Common infrastructure
  • Network: Centralized connectivity
  • Data Lake: Analytics and data science

Service Control Policies (SCPs)

  • Guardrails: Define maximum permissions for accounts
  • Inheritance: Policies flow down organizational hierarchy
  • Preventive Controls: Block actions before they happen
  • Compliance: Enforce regulatory and security requirements
  • Cost Control: Prevent expensive resource creation
  • JSON Format: Similar to IAM policies but focused on restrictions

Cross-Account Access Patterns

Cross-Account Roles:

  • Assume roles across account boundaries
  • Temporary credentials with time limits
  • Audit trail of cross-account access
  • Fine-grained permission control

Resource Sharing:

  • AWS Resource Access Manager (RAM)
  • VPC subnets and transit gateways
  • Route 53 resolver rules
  • License Manager configurations

Multi-Account Security Best Practices

  • Root Account Security: MFA and restricted access for all accounts
  • CloudTrail: Centralized logging across all accounts
  • Config Rules: Consistent compliance monitoring
  • GuardDuty: Threat detection across the organization
  • Security Hub: Centralized security findings aggregation
  • Access Analyzer: Review and validate access policies

Virtual Private Cloud (VPC)

Isolated network environment in AWS

VPC Overview

  • Logical Isolation: Private section of AWS cloud
  • Complete Control: Define IP address range, subnets, route tables
  • Security: Configure security groups and network ACLs
  • Connectivity: Connect to internet, other VPCs, or on-premises
  • Regional Service: Spans all Availability Zones in a region

Subnets

Public Subnets:

  • Route to Internet Gateway
  • Resources get public IP
  • Web servers, load balancers
  • Bastion hosts

Private Subnets:

  • No direct internet access
  • Route through NAT Gateway
  • Application servers
  • Database servers

VPC Security

Security Groups:

  • Instance-level firewall
  • Stateful filtering
  • Allow rules only
  • Default deny all inbound

Network ACLs:

  • Subnet-level firewall
  • Stateless filtering
  • Allow and deny rules
  • Default allow all

Internet Connectivity

  • Internet Gateway: Provides internet access to public subnets
  • NAT Gateway: Managed service for outbound internet from private subnets
  • NAT Instance: Self-managed EC2 instance for NAT functionality
  • Egress-Only Gateway: IPv6 outbound access only
  • VPC Endpoints: Private access to AWS services

VPC Connectivity Options

VPC-to-VPC:

  • VPC Peering: Direct connection between VPCs
  • Transit Gateway: Hub for multiple VPC connections
  • PrivateLink: Private connectivity to services
  • VPN Gateway: Encrypted tunnels

On-Premises:

  • AWS Direct Connect: Dedicated network connection
  • Site-to-Site VPN: Encrypted internet tunnel
  • Client VPN: Remote user access
  • Direct Connect Gateway: Multiple VPC access

VPC Design Patterns

  • Single VPC: Simple workloads with minimal isolation needs
  • Multi-VPC: Environment separation (dev, test, prod)
  • Hub and Spoke: Central hub VPC with spoke VPCs for workloads
  • Multi-Tier: Web, application, and database tiers in separate subnets
  • Shared Services VPC: Common services shared across environments
  • DMZ Pattern: Public-facing resources in dedicated VPC

VPC IP Address Planning

CIDR Best Practices:

  • Plan for growth and future requirements
  • Avoid overlapping with on-premises
  • Use /16 to /24 for VPC CIDR blocks
  • Reserve IP ranges for future use

Subnet Sizing:

  • AWS reserves 5 IPs per subnet
  • Plan for auto scaling requirements
  • Consider elastic network interfaces
  • Account for load balancer IPs

VPC Flow Logs

  • Traffic Monitoring: Capture IP traffic flow information
  • Security Analysis: Detect unusual traffic patterns
  • Troubleshooting: Diagnose connectivity issues
  • Compliance: Meet audit and compliance requirements
  • Destinations: CloudWatch Logs, S3, or Kinesis Data Firehose
  • Granularity: VPC, subnet, or network interface level

VPC Security Advanced Topics

Network Segmentation:

  • Micro-segmentation with security groups
  • Application-level isolation
  • Zero-trust network principles
  • Service mesh integration

Traffic Inspection:

  • AWS Network Firewall
  • Third-party firewall appliances
  • Intrusion detection systems
  • Deep packet inspection
