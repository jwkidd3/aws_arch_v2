Identity & Access Management (IAM)

Core Principles

  • Global Service: Applies across all regions
  • Principle of Least Privilege: Grant minimum required permissions
  • Defense in Depth: Multiple security layers

IAM Components

Core Identity Components

  • Users: Individual people or applications
  • Groups: Collections of users with similar access needs
  • Roles: Temporary credentials for services
  • Policies: JSON documents defining permissions

IAM Policy Types

Different Policy Categories

  • AWS Managed: Created and maintained by AWS
  • Customer Managed: Created and maintained by you
  • Inline: Directly attached to single entity

IAM Policy Evaluation

Policy Evaluation Logic

DENY always wins when conflicting permissions exist

Evaluation Process

  • Default: All requests denied by default
  • Explicit Allow: Must have explicit allow permission
  • Explicit Deny: Overrides any allow permissions

IAM Policy Structure

Element Required Description
Effect âœ“ Allow or Deny
Action âœ“ API calls to allow/deny
Resource âœ“ AWS resources affected

IAM Roles

When to Use Roles

  • EC2 Instances: Access AWS services without credentials
  • Cross-Account Access: Grant access to other accounts
  • AWS Services: Allow services to act on your behalf

Role Assumption Process

How Roles Work

Temporary credentials â†’ STS tokens â†’ Time-limited access

Security Benefit: No long-term credentials stored on instances

IAM Best Practices

Security Recommendations

  • Root Account: Never use for day-to-day operations
  • Individual Users: Create separate IAM users
  • Groups: Use groups to assign permissions

IAM Best Practices - Advanced

Advanced Security

  • MFA: Enable multi-factor authentication
  • Roles for Applications: Use roles vs embedded credentials
  • Regular Rotation: Rotate access keys regularly

Amazon Virtual Private Cloud (VPC)

VPC Fundamentals

  • Isolated Network: Your own section of AWS cloud
  • Complete Control: IP ranges, subnets, routing
  • Security: Multiple layers of security controls

VPC CIDR Planning

CIDR Notation

  • Range: /16 (65,536 IPs) to /28 (16 IPs)
  • Planning: Choose ranges that don't overlap
  • Growth: Allow room for future expansion
Best Practice: Use RFC 1918 private IP ranges
10.0.0.0/16, 172.16.0.0/16, 192.168.0.0/16

VPC Subnets

Subnet Types

  • Public Subnets: Direct route to Internet Gateway
  • Private Subnets: No direct internet access
  • AZ-Specific: Each subnet exists in one AZ
Design Rule: Always deploy across multiple AZs for high availability

VPC Gateways & Routing

Gateway Components

  • Internet Gateway: Connects VPC to internet
  • NAT Gateway: Outbound internet for private subnets
  • Route Tables: Control traffic routing decisions

Typical VPC Architecture

Three-Tier Architecture

Public Subnet (web) â†’ Private Subnet (app) â†’ Private Subnet (database)

Architecture Benefits

  • Security: Database tier isolated from internet
  • Scalability: Each tier scales independently

Security Groups vs NACLs

Feature Security Groups Network ACLs

Defense in Depth

Use both Security Groups and NACLs for layered security

VPC Endpoints

Endpoint Types

  • Gateway Endpoints: S3 and DynamoDB
  • Interface Endpoints: Most other AWS services
  • Benefit: Private connectivity without internet routing
Cost Optimization: VPC Endpoints reduce data transfer costs

Security Group Chaining

Tiered Security Pattern

Web Tier SG â†’ App Tier SG â†’ Database Tier SG

Chaining Benefits

  • Scalability: No IP address management required
  • Security: Each tier references previous tier's SG
